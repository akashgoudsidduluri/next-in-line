You are a senior backend systems engineer. Your task is to design and implement a production-grade internal hiring pipeline system that behaves like a deterministic state machine with queue semantics, not a CRUD app.

This system must be architected to maximize engineering quality signals (architecture, concurrency correctness, traceability, and development clarity), not just functional completeness.

---

# OBJECTIVE

Build a lightweight hiring pipeline tool for small teams that:

* Enforces a strict active capacity per job
* Automatically manages a waitlist queue
* Handles concurrency safely
* Automatically promotes candidates
* Handles inactivity decay with cascading effects
* Logs every state transition for full reconstruction

This system must run locally and be implemented using PERN or MERN (prefer PostgreSQL for correctness).

---

# CORE DESIGN REQUIREMENT (CRITICAL)

Treat the system as:

1. A **bounded active queue**
2. An **ordered waitlist queue**
3. A **finite state machine with event sourcing**

Every applicant must always be in exactly one state:

* WAITLISTED
* ACTIVE
* EXITED

State transitions must NEVER be implicit. Every transition must:

* Be atomic
* Be logged
* Be reproducible

---

# HARD REQUIREMENTS

## Queue + Capacity Logic

* Each job has a fixed `capacity`
* If active count < capacity → applicant becomes ACTIVE
* Else → applicant is WAITLISTED with a queue position
* Queue must be strictly ordered and gap-free

---

## Automatic Promotion

When an ACTIVE applicant exits:

* The first WAITLISTED applicant must be promoted automatically
* Promotion must:

  * Be transactional
  * Assign an acknowledgement deadline
  * Trigger further cascade logic if needed

---

## Concurrency Handling (MANDATORY)

Handle:
“Two applicants apply at the same time for the last available slot”

You MUST:

* Use database transactions
* Use row-level locking (`SELECT ... FOR UPDATE`)
* Prevent race conditions deterministically

Document:

* Why this approach works
* What would break without it

---

## Inactivity Decay (CORE DIFFERENTIATOR)

When a WAITLISTED applicant is promoted to ACTIVE:

* They must acknowledge within a defined time window (e.g., 10 minutes)

If they fail:

* They are NOT removed
* They are moved back to WAITLIST with a penalty position
* Next candidate is promoted immediately
* This must cascade automatically

You must define:

* Decay window
* Penalty strategy (e.g., +2 positions or end of queue)
* Trigger mechanism (polling loop or internal scheduler — NO external libraries allowed)

---

## Event Logging (NON-NEGOTIABLE)

Every transition must be logged in an `event_logs` table:

Each log must include:

* application_id
* event_type (APPLIED, PROMOTED, DECAYED, EXITED, ACKNOWLEDGED)
* metadata (JSON)
* timestamp

System must be reconstructable purely from logs.

Provide fully documented endpoints:

* Create job

* Acknowledge promotion
* Get applicant status (status + queue position + deadline)
* Get company dashboard (active + waitlist)

All responses must be deterministic and consistent.
